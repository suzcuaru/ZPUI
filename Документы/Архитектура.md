# Архитектура ZPUI

> Документ описывает техническую реализацию продукта: технологический стек, компонентную модель, потоки данных, жизненный цикл процессов, схему хранения состояния и сборочно-релизный конвейер. Продуктовый смысл — в [`Концепция.md`](./Концепция.md), доменные правила — в [`Бизнес-логика.md`](./Бизнес-логика.md), перечень возможностей — в [`Функции.md`](./Функции.md), визуальный язык — в [`Дизайн-и-типографика.md`](./Дизайн-и-типографика.md).

## 1. Технологический стек

### 1.1. Backend (Go)

| Компонент | Технология | Версия (по `go.mod`) | Назначение |
|---|---|---|---|
| Язык | Go | 1.21+ | основная логика, все сетевые сервисы |
| Каркас приложения | **Wails v2** | v2.x | связка Go ↔ web-фронтенд; нативное окно с WebView2 на Windows |
| База данных | **modernc.org/sqlite** | — | чисто-Go драйвер SQLite (без cgo), хранит историю мониторинга и настройки |
| Логирование | `log/slog` (stdlib) + собственный `logger` | — | структурные логи в файл |
| HTTP-клиент | stdlib `net/http` | — | скачивание zapret, обновлений, проверка ресурсов |
| Сеть/DNS | stdlib `net`, `golang.org/x/net/dns/dnsmessage` | — | внутренний DNS-сервер (Xbox DNS) |
| SOCKS5 | собственная реализация в `internal/proxy` | — | раздача обхода на другие устройства |
| WinDivert-обёртка | вызов `winws.exe` (zapret) как дочернего процесса | — | движок обхода DPI |
| Трей/уведомления | Win32 API (`syscall`, `golang.org/x/sys/windows`) | — | иконка в системном трее, balloon-уведомления |
| Реестр Windows | `golang.org/x/sys/windows/registry` | — | автозапуск, ассоциации |
| Одноэкземплярность | `internal/singleinstance` (named mutex) | — | гарантия, что запущена одна копия ZPUI |

**Почему Wails, а не Electron / Tauri / чистый WinForms**

- **Electron** — тяжелый (Chromium + Node), ~150 МБ бинарник. Для утилиты «скачал-запустил» — перебор.
- **Tauri** — Rust + WebView, классный, но стек проекта — Go, а не Rust; переписывать backend нецелесообразно.
- **WinForms / WPF** — нативно для Windows, но UI рисуется кодом, медленно итерировать дизайн; нет реактивности.
- **Wails** — Go backend (вся логика на знакомом языке) + любой web-фронтенд (React, Svelte, Vue) в нативном WebView2. Размер бинарника ~10 МБ. Оптимально для десктоп-утилиты.

### 1.2. Frontend (web/)

| Компонент | Технология | Версия (по `web/package.json`) | Назначение |
|---|---|---|---|
| Каркас UI | **React 18** | 18.x | компонентная модель |
| Сборщик | **Vite** | 5.x | дев-сервер, production-бандл |
| Маршрутизация | **React Router v6** | — | навигация между вкладками |
| i18n | собственная обёртка `useT` + `locales/{ru,en}.json` | — | локализация |
| Стили | чистый CSS + CSS Custom Properties (переменные) | — | дизайн-система без Tailwind |
| Иконки | inline SVG-компоненты | — | векторная графика |
| Графики | легковесные canvas/SVG-компоненты | — | дашборд мониторинга |
| Тесты | **Vitest** | — | unit-тесты утилит и хуков |

**Почему чистый CSS с переменными, а не Tailwind / CSS-in-JS**

- Tailwind раздувает HTML utility-классами, затрудняет тематирование с-runtime-параметрами (у ZPUI динамические цвета из настроек).
- CSS-in-JS (styled-components, emotion) добавляет рантайм-накладные.
- CSS Custom Properties (`--bg`, `--text`, `--accent`) нативно поддерживаются, переключаются в рантайме (тёмная/светлая темы, акцентный цвет), не требуют сборщика.

### 1.3. Инструментарий сборки и релиза

| Компонент | Технология | Назначение |
|---|---|---|
| Инсталлятор | **NSIS** (Nullsoft Scriptable Install System) | `.exe`-установщик для конечного пользователя (`installer/ZPUI.nsi`) |
| CI/CD | **GitHub Actions** | `.github/workflows/ci.yml` (сборка/тесты), `.github/workflows/release.yml` (публикация релиза) |
| Локальная сборка | `build.bat` | оркестрация: bump версии → build frontend → build Wails core → build 4 модулей → упаковка в `build/dist/` |
| Релиз-скрипт | `release.bat` | интерактивный помощник релиза (tag, push, заметки) |
| Git-менеджер | `git-manager.bat` | вспомогательный скрипт для рутинных git-операций |
| Зеркало | **Яндекс.Диск** | резервный канал распространения, primary-источник для satellite-обновлятора |

## 4. Слой бэкенда

### 4.1. Точка входа `main.go`

Минимальный: создаёт экземпляр `app.App` и передаёт его в `wails.Run()` с конфигурацией окна. Вся логика — в `internal/app`.

```go
// упрощённо
func main() {
    application := app.NewApp()
    err := wails.Run(&options.App{
        Title:     "ZPUI",
        Width:     960,
        Height:    640,
        AssetServer: ...,
        OnStartup: application.OnStartup,
        OnShutdown: application.OnShutdown,
        Bind: []interface{}{application},
    })
}
```

### 4.2. Ядро `internal/app`

**`App` struct** — центральный объект. Все его методы (`func (a *App) ...`) автоматически становятся доступны фронтенду как `window.go.app.App.*`.

Жизненный цикл:

1. `NewApp()` — конструктор, инициализация полей-зависимостей.
2. `OnStartup(ctx)` — вызывается Wails при старте. Здесь:
   - проверяется single-instance (named mutex);
   - открывается/создаётся SQLite, config.json;
   - инициализируется logger (открываются файлы в `logs/`);
   - создаётся иконка трея;
   - запускаются фоновые горутины (monitor polling, autostart check).
3. Бинд-методы обрабатывают запросы от фронтенда (синхронные или через `runtime.EventsEmit` для асинхронной передачи прогресса).
4. `OnShutdown(ctx)` — корректное завершение: останавливается winws.exe, закрываются серверы (SOCKS5, DNS), flush логов, освобождается mutex.

**Группировка биндингов** (`app_*.go`):

| Файл | Группа методов | Что делает |
|---|---|---|
| `app_zapret.go` | `ZapretStart`, `ZapretStop`, `ZapretStatus`, ... | управление движком |
| `app_strategies.go` | `ListStrategies`, `SetStrategy`, ... | выбор пресета |
| `app_autoselect.go` | `RunAutoselect`, `AutoselectStatus` | автоподбор |
| `app_proxy.go` | `ProxyStart`, `ProxyStop`, `ProxyStatus` | SOCKS5 |
| `app_xboxdns.go` | `DnsStart`, `DnsStop`, ... | Xbox DNS |
| `app_monitor.go` | `MonitorStatus`, `MonitorHistory` | мониторинг |
| `app_blockcheck.go` | `CheckBlock`, ... | проверка блокировки |
| `app_updater.go` | `CheckUpdates`, `ApplyUpdate`, ... | обновления |
| `app_sysinfo.go` | `SysInfo`, `NetInfo`, ... | сведения о системе |
| `app_config.go` | `GetConfig`, `SetConfig` | настройки |
| `app_wizard.go` | `RunWizard`, ... | первый запуск |
| `app_versions.go` / `versions.go` | `GetVersions` | версии ZPUI и модулей |

**Ответы API** — единый формат через хелперы из `app_api_types.go`:

- `okResp(data)` → `{"status":"ok", ...data}`
- `errResp(msg)` → `{"error":"..."}`
- фронтенд проверяет наличие поля `error`.

### 4.3. Управление zapret (`internal/zapret`)

Движок — внешний процесс `winws.exe` (zapret). ZPUI им управляет, а не запускает DPI-обход сам.

- **`manager.go`** — высокоуровневое управление: старт/стоп, отслеживание PID, перезапуск при падении, восстановление после сбоя. Хранит активную стратегию, аргументы командной строки.
- **`service.go`** — обёртка над Windows Service API: zapret можно зарегистрировать как системную службу (чтобы работал до входа пользователя в систему и при перезагрузках). Здесь — установка/удаление службы, старт/стоп через Service Control Manager.

Поток запуска:

```
App.ZapretStart(strategy)
   └─ manager.Start(ctx, strategy)
        ├─ собрать CLI-аргументы из стратегии (фильтр-списки, опции)
        ├─ executil.Run("winws.exe", args...)
        ├─ сохранить PID, подписаться на процесс (гérdина wait)
        └─ вернуть статус во фронтенд
```

При падении winws.exe (краш, OOM, антивирус) `manager` перехватывает событие, инкрементит счётчик рестартов и поднимает заново, если не превышен лимит.

### 4.4. Автоподбор стратегии (`internal/autoselect`)

Перебирает пресеты zapret, для каждого:

1. Запускает winws.exe с аргументами стратегии.
2. Ждёт стабилизации (2–5 сек).
3. Через `blockcheck` проверяет доступность целевых ресурсов (discord.com, youtube.com, ...).
4. Останавливает, переходит к следующей.
5. По итогу выбирает стратегию с максимальным % доступности.
6. Применяет её как активную.

Прогресс пушится во фронтенд через `runtime.EventsEmit("autoselect:progress", {...})`.

### 4.5. Мастер первого запуска (`internal/wizard`)

Последовательность шагов:

1. Проверка системы (`sysinfo`): версия Windows, разрядность, права админа, антивирус.
2. Проверка наличия zapret: если нет — скачивание и распаковка (`updater`).
3. Проверка целостности файлов (SHA-256 манифест).
4. Запуск `autoselect` (первичный подбор).
5. Применение найденной стратегии, тест.
6. Запись результата в `config.json` (wizard.completed = true).

### 4.6. SOCKS5-прокси (`internal/proxy`)

Внутренний сервер на Go (stdlib `net`):

- слушает на настраиваемом порту (по умолчанию 1080);
- поддерживает авторизацию по логину/паролю (опционально);
- заворачивает исходящий трафик клиентов (телефонов, планшетов) в соединения, которые проходят через zapret (потому что winws.exe перехватывает на уровне драйвера все исходящие пакеты машины);
- даёт клиентам «обход» без установки zapret на них самих.

### 4.7. Xbox DNS (`internal/xboxdns`)

Внутренний DNS-сервер на Go:

- слушает UDP:53 (требует прав админа / reservation);
- отвечает на запросы A/AAAA для доменов Xbox;
- подменяет/фильтрует ответы так, чтобы консоль шла на правильные адреса (подробности — закрытый but специфичный для Xbox Live блокировок);
- позволяет указать IP компьютера как DNS-сервер на консоли.

### 4.8. Мониторинг (`internal/monitor`)

Фоновая горутина:

- раз в N секунд (настраиваемый интервал) проверяет доступность пула ресурсов через HTTP(S)-запросы и TCP-коннекты;
- результат пишется в SQLite (история для графиков) и пушится во фронтенд через события;
- на основе метрики доступности может триггерить **AutoSwitch** (см. [`Бизнес-логика.md`](./Бизнес-логика.md)).

### 4.9. Проверка блокировок (`internal/blockcheck`)

Утилита, отвечающая на вопрос «заблокирован ли ресурс X прямо сейчас?»:

- TCP-коннект к хосту:443 с таймаутом;
- TLS-рукопожатие (проверка «оборвалось ли»);
- HTTP-запрос с анализом кода/тела;
- сравнение поведения с эталоном (прямое подключение vs через потенциальный DPI).

### 4.10. Обновления (`internal/updater`)

Два независимых направления:

- **Обновление zapret:** качает свежий архив с GitHub (`Flowseal/zapret-discord-youtube` releases), распаковывает, перед применением делает резервную копию текущего, проверяет SHA-256, заменяет, перезапускает движок.
- **Обновление ZPUI:** отдельный модуль `cmd/selfupdate` качает zip с Яндекс.Диска (primary) или GitHub (fallback), проверяет версию, запускает замену в «холодном» режиме (закрывает ZPUI, подменяет бинарник, перезапускает).

### 4.11. Конфигурация (`internal/config`)

`config.json` хранит:

- активную стратегию zapret и её параметры;
- состояние тумблеров (прокси вкл/выкл, DNS вкл/выкл, автозапуск, AutoSwitch);
- настройки темы (акцентный цвет, тёмная/светлая);
- параметры мониторинга (интервал, порог AutoSwitch, список ресурсов);
- флаги мастера первого запуска;
- версии компонентов (для отображения в UI).

Загрузка — ленивая, с дефолтами; сохранение — через `useDebouncedSave` во фронтенде (предотвращает лишние записи при дрг-н-дропе слайдеров).

### 4.12. База данных (`internal/database`)

SQLite (pure-Go через `modernc.org/sqlite`):

- **history** — временные ряды доступности ресурсов (для графиков мониторинга);
- **stats** — агрегаты за день/неделю;
- **события** — лог ключевых действий (старт/стоп zapret, смена стратегии, срабатывание AutoSwitch);
- чистка старых записей — по расписанию.

### 4.13. Трей (`internal/tray`)

- иконка в системном трее Windows;
- контекстное меню: «Показать», «Старт/Стоп», «Выход»;
- balloon-уведомления о ключевых событиях (запустился, стратегия переключиласьAutoSwitch'ем, обновление готово).

### 4.14. Одноэкземплярность (`internal/singleinstance`)

Через Win32 named mutex: если ZPUI уже запущен, второй экземпляр обнаруживает занятый mutex, активирует окно первого и завершается.

### 4.15. Модули-расширения (`internal/mods`)

Точка расширения: плагины-модификаторы, которые могут добавлять фильтры или поведенческие модули без правки ядра (например, game-filter для онлайн-игр).

## 5. Слой фронтенда

### 5.1. Точка входа и роутинг

- `index.html` — пустой каркас + `<div id="root">`.
- `src/main.jsx` — монтирует `<App />` в root.
- `src/App.jsx` — корневой компонент с `<BrowserRouter>` и `<Routes>`.

Основные экраны-вкладки:

| Путь | Экран |
|---|---|
| `/` | Дашборд (обзор, статус, графики) |
| `/strategies` | Стратегии zapret |
| `/lists` | Редактор списков доменов |
| `/proxy` | SOCKS5-прокси |
| `/dns` | Xbox DNS |
| `/monitor` | Монитор доступности |
| `/diagnostics` | Диагностика системы |
| `/updates` | Обновления |
| `/settings` | Настройки |
| `/wizard` | Мастер первого запуска (оверлей) |

### 5.2. Шим API (`src/api.js`)

Фронтенд НЕ делает HTTP-запросов к бэкенду (Wails работает в одном процессе, без встроенного HTTP-сервера). Вместо этого — единая функция-шим:

```js
// упрощённо
export async function api(method, route, body) {
    const fn = routeToBinding(route); // '/api/zapret/start' → app.ZapretStart
    return await fn(body);
}
```

`routeToBinding` — карта маршрутов: строка вида `GET /api/...` сопоставляется с конкретным `window.go.app.App.<Method>`. Все эндпоинты бэкенда регистрируются здесь же (см. [`AGENTS.md`](../AGENTS.md)).

**Преимущества:** единая точка для добавления эндпоинтов; одинаковая обработка ошибок; фронтенду не важно, HTTP это или IPC.

### 5.3. Хуки (`src/hooks/`)

| Хук | Назначение |
|---|---|
| `usePolling(fn, interval, deps)` | периодический вызов `fn` (пинг статуса), автоочистка по unmount |
| `useDebouncedSave(value, saveFn, delay)` | отложенное сохранение (ползунки, текстовые поля) — не пишет на каждый чих |
| `useServiceToggle(serviceName)` | инкапсулирует логику тумблера «старт/стоп сервиса» с optimistic UI и откатом при ошибке |

### 5.4. UI-компоненты (`src/components/ui/`)

Базовые кирпичики:

- `Row` — строка-настройка (иконка + подпись + контрол).
- `Switch` — тумблер.
- `Modal` — модальное окно.
- `Button`, `Card`, `Badge`, `Slider`, `Select`, `Tabs`, `Toast`, `Tooltip`, ...

Все — на CSS Custom Properties, без жёстко зашитых цветов (см. [`Дизайн-и-типографика.md`](./Дизайн-и-типографика.md)).

### 5.5. i18n (`src/i18n.js`, `src/locales/`)

- `useT()` — React-хук, возвращает функцию-переводчик с текущей локалью.
- `tr(key, lang?)` — не-React-доступ (для утилит).
- Все пользовательские строки в UI — через `t('...')`.
- Локали: `ru.json` (основная), `en.json` (опциональная).

### 5.6. Асинхронное взаимодействие с бэкендом

Помимо прямых вызовов биндингов, фронтенд подписывается на **события Wails** (`runtime.EventsOn`):

- `autoselect:progress` — прогресс автоподбора;
- `monitor:update` — новый сэмпл мониторинга;
- `wizard:step` — переход шага мастера;
- `update:available` — найдено обновление;
- `toast:show` — показать уведомление.

Это позволяет бэкенду пушить прогресс долгих операций без поллинга со стороны UI.

## 6. Схема потоков данных

### 6.1. Поток «пользователь нажал Старт на дашборде»

```
[User click]
   └─ React: api('POST','/api/zapret/start', {strategy})
        └─ api.js: app.ZapretStart({strategy})
             └─ App.ZapretStart(payload) [Go, bound method]
                  └─ zapret.Manager.Start(ctx, payload.strategy)
                       ├─ сбор CLI-аргументов
                       ├─ executil.Run("winws.exe", args...)
                       └─ return {status:"ok", pid:...}
   └─ React: обновляет UI (optimistic), показывает статус «Запущен»
   └─ Фоново: monitor продолжает поллинг, winws.exe модифицирует пакеты
```

### 6.2. Поток «автоподбор стратегии»

```
[User click "Автовыбор"]
   └─ api: app.RunAutoselect()
        └─ autoselect.Run(ctx)
             ├─ for each strategy:
             │    ├─ zapret.Manager.Start(strategy)
             │    ├─ EventsEmit("autoselect:progress", {current, total, strategy})
             │    ├─ blockcheck.Check(resources)
             │    ├─ сохранить результат
             │    └─ zapret.Manager.Stop()
             ├─ выбрать best
             ├─ zapret.Manager.Start(best)  // применить
             └─ config.SetActiveStrategy(best)
   └─ Frontend (EventsOn "autoselect:progress"): обновляет прогресс-бар
   └─ Frontend: показывает результат, переключает карточку активной стратегии
```

### 6.3. Поток «мониторинг и AutoSwitch»

```
[OnStartup горутина monitor]
   └─ каждые N сек:
        ├─ blockcheck.Check(allResources)
        ├─ database.WriteHistory(samples)
        ├─ EventsEmit("monitor:update", samples)
        └─ if availability < threshold && AutoSwitch on:
             ├─ autoselect.Run() (фон, с уведомлением)
             └─ tray.Notify("AutoSwitch: переключился на <strategy>")
```

### 6.4. Поток «обновление zapret»

```
[User click "Проверить обновления"]
   └─ api: app.CheckUpdates()
        └─ updater.CheckZapret()
             ├─ HTTP GET github.com/.../releases/latest
             ├─ сравнить версию с config
             └─ return {hasUpdate, version, url}
   └─ Frontend: показывает «Доступно vX.Y.Z»
[User click "Обновить"]
   └─ api: app.ApplyUpdate()
        └─ updater.DownloadAndApply()
             ├─ zapret.Manager.Stop()
             ├─ backup текущего в zapret.bak/
             ├─ HTTP GET zip
             ├─ SHA-256 проверка
             ├─ распаковать в zapret/
             ├─ zapret.Manager.Start() (со старой стратегией)
             └─ tray.Notify("zapret обновлён")
```

### 6.5. Поток «телефон через SOCKS5»

```
[Phone Wi-Fi settings]
   └─ Phone → socks5://192.168.1.10:1080
        └─ proxy.Server (внутри ZPUI.exe)
             ├─ accept соединение
             ├─ парсим SOCKS5 CONNECT host:port
             ├─ net.Dial("tcp", host:port)
             │    ↑ этот dial перехватывается WinDivert (winws.exe),
             │      модифицируются пакеты → DPI пропускает
             └─ bidirectional copy phone ↔ target
```

## 7. Управление состоянием

| Слой | Где | Что хранит |
|---|---|---|
| Время сессии (runtime) | `App` struct поля | PID zapret, активные серверы, подписки |
| Пользовательские настройки | `config.json` | стратегии, тумблеры, тема, интервалы |
| История и статистика | `SQLite` (`zappy.db`) | временные ряды мониторинга, события |
| Логи | `logs/*.log` | структурные slog-записи (ротация по дате) |
| Состояние UI | React state + hooks | текущая вкладка, формы, optimistic-флаги |

Принцип: **единственный источник истины для настроек — `config.json`**. UI — отражение этого состояния; изменение в UI → запись в config → реактивное обновление. Это гарантирует, что после перезапуска всё восстанавливается.

## 8. Жизненный цикл процессов

### 8.1. Запуск ZPUI.exe

1. `main.go` → `app.NewApp()`.
2. `wails.Run()` → инициализация WebView2.
3. `OnStartup(ctx)`:
   - `singleinstance.Acquire()` — если занято, активировать существующее окно и выйти.
   - `logger.Open()` — открыть файлы логов.
   - `config.Load()` — прочитать `config.json`.
   - `database.Open()` — открыть SQLite.
   - `tray.Create()` — иконка + меню.
   - Запуск фоновых горутин: `monitor`, `autostart.Ensure()`.
   - Если `config.wizard.completed == false` — фронтенд открывает мастер.
4. WebView2 грузит собранный фронтенд (`web/dist`).
5. Фронтенд вызывает `app.GetConfig()`, `app.MonitorStatus()`, ... — строит UI.

### 8.2. Запуск zapret

- По требованию пользователя (кнопка «Старт») ИЛИ автоматически, если в config включён автозапуск движка.
- `winws.exe` запускается как дочерний процесс ZPUI.exe; если ZPUI закрывается — winws тоже (если не зарегистрирован как служба).

### 8.3. Завершение

- `OnShutdown(ctx)`:
  - `zapret.Manager.Stop()` — корректно завершить winws.exe.
  - `proxy.Server.Stop()`, `xboxdns.Server.Stop()`.
  - `database.Close()`, `logger.Close()`.
  - `singleinstance.Release()`.
- Если zapret установлен как служба — он продолжает работать и после закрытия UI.

## 9. Безопасность

- **Права администратора** требуются для: запуска `winws.exe` (драйвер WinDivert), прослушивания DNS на :53, изменения автозапуска в реестре. Манифест приложения (`installer/ZPUI.nsi`) запрашивает повышение через UAC.
- **Проверка целостности:** скачиваемые архивы сверяются по SHA-256 с манифестом (на Яндекс.Диске/GitHub).
- **Логи не содержат чувствительных данных:** пароли прокси не пишутся в открытый вид; IP-адреса пользователей не отправляются наружу.
- **Нет телеметрии:** ZPUI не отчитывается никуда о пользователях, ошибках, версиях. Все сетевые запросы — только к GitHub и Яндекс.Диску за самим zapret/обновлениями.
- **Open-source (MIT):** любой может аудитнуть код.

## 10. Сборка и релиз

### 10.1. Локальная сборка (`build.bat`)

Последовательно:

1. Чтение и bump версии ZPUI из `version.txt` (minor или patch — по аргументу).
2. `cd web && npm install && npm run build` — production-бандл фронтенда.
3. `wails build` — сборка `ZPUI.exe` (встраивает `web/dist` через `embed.FS`).
4. Сборка 4 модулей:
   - `cd cmd/wizard && go build -o ... wizard.exe`
   - `cd cmd/autoselect && go build ...`
   - `cd cmd/selfupdate && go build ...`
   - `cd cmd/zapretupdate && go build ...`
5. Формирование `versions.json` (читает `var version` из каждого `cmd/*/main.go`).
6. Упаковка в `build/dist/`: `ZPUI.exe`, модули, zapret, `versions.json`, `config.json` (дефолт).
7. NSIS-компиляция установщика: `makensis installer/ZPUI.nsi` → `ZPUI-Setup-x.y.z.exe`.

### 10.2. Версионирование

- **ZPUI core** — SemVer в `version.txt`, бампится `build.bat`.
- **4 модуля** — каждый имеет свой `var version = "1.0.0"` в `cmd/*/main.go`. **НЕ** бампятся автоматически — ручное изменение при правках модуля.
- `versions.json` — собираемый артефакт с версиями всех компонентов (для UI «О программе» и для checker'а обновлений).

### 10.3. CI (`ci.yml`, windows-latest)

На каждый push/PR:

- `go vet ./...`
- `go test ./...`
- `cd web && npm install && npm run build`
- (опционально) `go build ./...`

### 10.4. Релиз (`release.yml`, ubuntu-latest)

**Важно:** этот job НЕ собирает бинарники (сборка Windows-нативная через Wails + NSIS, на Ubuntu невозможна). Он:

1. Реагирует на tag `v*.*.*`.
2. Берёт артефакты из `build/dist/` (залитые вручную или через отдельный Windows-runner).
3. Формирует GitHub Release с описанием из `CHANGELOG`/коммитов.
4. Загружает установщик и zip на **Яндекс.Диск** (через WebDAV API) как зеркало.

Подробно — в [`AGENTS.md`](../AGENTS.md) и `GIT_WORKFLOW.md`.

## 11. Кроссплатформенные соображения

Текущая поддержка — **только Windows 10/11 (x86 и x64)**. Причины:

- WinDivert — Windows-only.
- Wails v2 поддерживает macOS/Linux, но движок zapret под них — другой (nfqueue на Linux, pf на macOS), что потребовало бы отдельного backend-слоя.
- Трей, реестр, named mutex — Win32 API.

Портирование не планируется (см. [`Концепция.md`](./Концепция.md), §9.2).

## 12. Производительность и ограничения

- **Память:** ZPUI.exe ~80–120 МБ (WebView2 + Go runtime). winws.exe добавляет 10–30 МБ.
- **CPU:** в простое ~0%, при активном обходе — зависит от объёма трафика (DPI-модификация делается в ядре, эффективно).
- **Диск:** ~30 МБ бинарники + логи (с ротацией) + SQLite-история (чистится).
- **Сеть:** все пользовательские соединения проходят через WinDivert; дополнительные задержки — на фрагментацию TLS ClientHello (~миллисекунды).
