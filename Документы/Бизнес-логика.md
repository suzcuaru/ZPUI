# Бизнес-логика ZPUI

> Документ описывает доменные правила, состояния, сценарии и алгоритмы принятия решений в ZPUI. Это НЕ техническая архитектура компонентов (см. [`Архитектура.md`](./Архитектура.md)) и НЕ список кнопок (см. [`Функции.md`](./Функции.md)), а **правила поведения продукта**: в каких состояниях может находиться система, при каких условиях происходят переходы, как выбирается стратегия, как принимается решение «работает/не работает».

---

## 1. Доменные сущности

| Сущность | Описание | Где хранится |
|---|---|---|
| **Стратегия (Strategy / Preset)** | Именованный набор аргументов для `winws.exe`: фильтр-список, опции фрагментации, fake-пакеты и т. п. | `zapret/strategies/*` или код |
| **Ресурс (Resource)** | Цель проверки доступности: `discord.com`, `youtube.com`, `discord.gg`, и т. д. | `config.resources[]` |
| **Состояние движка (EngineState)** | `stopped` / `starting` / `running` / `error` / `restarting` | runtime App |
| **Доступность (Availability)** | Процент ресурсов из пула, доступных в данный момент (0–100%) | computed из `blockcheck` |
| **Порог AutoSwitch (Threshold)** | Значение доступности, ниже которого триггерится автопереключение | `config.autoswitch.threshold` |
| **Профиль пользователя** | Набор предпочтений: тема, язык, интервал мониторинга, автозапуск, уведомления | `config.json` |
| **Событие (Event)** | Записываемое действие: старт zapret, смена стратегии, обновление, срабатывание AutoSwitch | SQLite `events` |

## 2. Состояния движка zapret

```
                 ┌──────────┐
                 │ stopped  │◄──────────────────────────┐
                 └────┬─────┘                            │
                      │ user: Start                      │
                      ▼                                  │
                 ┌──────────┐   error (timeout/      ┌───┴──────┐
                 │ starting │───crash/antivirus)────►│  error   │
                 └────┬─────┘                         └────┬─────┘
                      │ winws ready                         │ user: retry
                      ▼                                     │
                 ┌──────────┐    crash (with restart   ┌────┴──────┐
        ┌───────►│ running  │◄───budget left)──────── │restarting │
        │        └────┬─────┘                         └───────────┘
        │             │ user: Stop / shutdown              ▲
        │             │                                     │
        └─────────────┘                                     │
   auto-restart success                            crash detected
```

### Правила переходов

- **`stopped → starting`**: пользователь нажал «Старт», ИЛИ AutoSwitch инициировал, ИЛИ wizard применил стратегию, ИЛИ автозапуск при старте системы.
- **`starting → running`**: `winws.exe` запущен, процесс жив > N секунд (стабилизация), драйвер WinDivert загрузился.
- **`starting → error`**: процесс упал в первые секунды, WinDivert недоступен (антивирус удалил), отсутствие прав админа.
- **`running → restarting`**: процесс упал, счётчик рестартов < лимита (по умолчанию 5 за 60 сек).
- **`restarting → running`**: успешный рестарт.
- **`restarting → error`**: лимит рестартов исчерпан — считается устойчивый сбой.
- **`running → stopped`**: пользователь нажал «Стоп», ИЛИ приложение завершается.
- **`error → starting`**: пользователь нажал «Повторить», ИЛИ запущена диагностика и проблема устранена.

### Счётчик рестартов

- Окно: 60 секунд. Если за это время произошло ≥ 5 рестартов — движок переходит в `error`, пользователю показывается тост с рекомендацией открыть диагностику.
- Счётчик сбрасывается, если движок проработал стабильно > 60 сек.

## 3. Выбор стратегии

### 3.1. Ручной выбор

Пользователь из вкладки «Стратегии» выбирает карточку → `SetStrategy(id)` → движок перезапускается с новыми аргументами → стратегия пишется в `config.activeStrategy` как активная.

### 3.2. Автоподбор (autoselect)

**Цель:** автоматически найти стратегию с максимальной доступностью для текущего провайдера/DPI.

**Алгоритм:**

1. Собрать список кандидатов (все стратегии, или подмножество по фильтру — например, только «новые» или «универсальные»).
2. Запомнить текущую стратегию (для отката).
3. Для каждой стратегии по очереди:
   a. Остановить движок (если запущен).
   b. Применить стратегию, запустить `winws.exe`.
   c. Ждать стабилизации (по умолчанию 5 сек).
   d. Запустить `blockcheck` по пулу ресурсов.
   e. Вычислить % доступности.
   f. Запомнить результат.
   g. Остановить движок.
   h. Пушить прогресс через `EventsEmit("autoselect:progress")`.
4. Выбрать стратегию с максимальным %.
5. Если у лучшего кандидата доступность ≥ «приемлемого минимума» (по умолчанию 70%) — применить её как активную.
6. Если все стратегии дали < минимума — применить лучшую из худших, уведомить пользователя, что DPI агрессивный и нужен ручной тюнинг или обновление zapret.
7. Запись события в БД: `{type:"autoselect", winner:strategyId, bestAvailability:N}`.

**Приоритезация кандидатов** (эвристика, ускоряющая перебор):

1. Последняя успешная стратегия (из истории).
2. Стратегии, помеченные как «универсальные».
3. Стратегии с наибольшей средней исторической доступностью.
4. Остальные.

**Прерывание:** пользователь может отменить автоподбор в любой момент. Текущая тестируемая стратегия останавливается, восстанавливается предыдущая активная.

### 3.3. AutoSwitch (фоновое самоизлечение)

**Цель:** без участия пользователя реагировать на деградацию обхода.

**Параметры (в `config.autoswitch`):**

- `enabled` — вкл/выкл.
- `threshold` — порог доступности, ниже которого срабатывает (по умолчанию 60%).
- `cooldown` — минимальный интервал между срабатываниями (по умолчанию 10 минут), чтобы не «молотил».
- `strategyPool` — список стратегий для перебора (по умолчанию все).
- `consecutiveFails` — сколько подряд сэмплов ниже порога триггерят (по умолчанию 2, защита от единичных замеров).

**Алгоритм срабатывания:**

1. Monitor каждые N сек замеряет доступность.
2. Если `availability < threshold` — инкремент `failCounter`.
3. Если `failCounter ≥ consecutiveFails` И `time.Since(lastSwitch) ≥ cooldown`:
   a. Запустить `autoselect.Run()` на подмножестве `strategyPool`.
   b. Если найдена стратегия лучше текущей — переключиться.
   c. `tray.Notify("AutoSwitch: переключился на <strategy>")`.
   d. Сбросить `failCounter`, обновить `lastSwitch`.
4. Если доступность восстановилась — `failCounter` сбрасывается.

**Гарантия:** AutoSwitch НЕ отключает пользователя полностью. В худшем случае он переключает на лучшую из доступных стратегий; даже если ни одна не даёт 100%, пользователю обеспечивается максимум возможного.

## 4. Оценка доступности ресурсов

### 4.1. Что считается «ресурс доступен»

Ресурс считается доступным, если выполняется **комбинированный критерий**:

1. **TCP-коннект** к `host:443` успешен в течение таймаута (3 сек).
2. **TLS-рукопожатие** завершается без alert/RST.
3. **HTTP(S)-запрос** `/` возвращает ожидаемый код (200/301/302/4xx, не 0/5xx-от-провайдера).

Все три проверки должны пройти. Если любая падает — ресурс в этом сэмпле недоступен.

### 4.2. Расчёт процента

```
availability = (количество доступных ресурсов / всего ресурсов в пуле) * 100
```

С гистерезисом: единичный «провальный» сэмпл не считается мгновенным падением — используется скользящее окно из 3 сэмплов (медиана) для UI, чтобы дашборд не «дёргался».

### 4.3. Таймауты и обработка ошибок

- Каждая проверка идёт с таймаутом 3 сек (через `ctx`).
- Параллельные проверки ресурсов (`go func` для каждого).
- Сетевые ошибки (DNS, connection refused, TLS alert) → ресурс недоступен.
- Таймаут → ресурс недоступен (вероятно DPI глушит).

### 4.4. Пул ресурсов

Настраивается в `config.resources`. По умолчанию включает ключевые для продукта цели:

- `discord.com`, `discord.gg`, `discordapp.com`, `discordapp.net`
- `youtube.com`, `googlevideo.com`, `ytimg.com`
- (опционально) `twitch.tv`, `instagram.com`, пользовательские домены

## 5. SOCKS5-прокси: правила работы

### 5.1. Когда слушать

- Прокси-сервер стартует, **только** если пользователь включил тумблер «Прокси».
- Порт по умолчанию 1080; можно изменить.
- При смене порта сервер перезапускается.

### 5.2. Авторизация

- Без авторизации (по умолчанию) — доступ из локальной сети.
- С авторизацией (login + password) — рекомендуется, если ПК виден из недоверенной сети.

### 5.3. Что проксируется

- SOCKS5 CONNECT на любой host:port — обычное поведение SOCKS5.
- Прокси НЕ фильтрует контент, НЕ блокирует сайты, НЕ подменяет адреса.
- Весь исходящий трафик от клиентов идёт через сокеты ZPUI, которые перехватываются WinDivert (winws.exe) на уровне драйвера — поэтому клиенты получают обход без установки zapret.

### 5.4. Ограничения

- Только TCP (SOCKS5 UDP ASSOCIATE поддерживается ограниченно).
- Не работает для клиентов, которые сами используют VPN поверх SOCKS (двойной обход может конфликтовать).
- Производительность ограничена пропускной способностью ПК-хозяина.

## 6. Xbox DNS: правила работы

### 6.1. Когда слушать

- DNS-сервер стартует при включении тумблера «DNS».
- Слушает UDP:53 (требует прав админа и чтобы порт не занят другим DNS-сервисом).

### 6.2. Логика ответов

- Для доменов Xbox Live — отвечает настроенными A-записями.
- Для остальных доменов — либо проксирует upstream (системный DNS), либо отвечает REFUSED (зависит от режима).
- Режимы: «только Xbox» (минимум вмешательства) / «полный резолвер» (вся DNS-карта консоли через ZPUI).

### 6.3. Конфликт порта 53

- Если порт занят (Windows DNS Client, другой резолвер) — запуск падает с понятной ошибкой; пользователь получает инструкцию (отключить DNS Client или выбрать другой режим).

## 7. Мастер первого запуска (Wizard): бизнес-правила

Последовательность шагов и условия перехода:

### Шаг 1. Проверка системы

- ОС: Windows 10/11? Если нет — предупреждение, работа возможна, но не гарантируется.
- Разрядность: определяется x86/x64 (для выбора варианта zapret).
- Права админа: если нет — нельзя продолжать (WinDivert требует). Показать инструкцию.
- Антивирус: если детектируется активный AV — предупреждение «добавьте ZPUI в исключения, иначе драйвер удалится».

### Шаг 2. Установка zapret

- Если zapret уже присутствует и целостность в норме — пропуск шага.
- Если нет — скачать с зеркала (GitHub primary, Диск fallback).
- Проверка SHA-256 манифеста.
- Распаковка в `zapret/`.
- Проверка целостности.

### Шаг 3. Первичный автоподбор

- Запускается `autoselect.Run()` на базовом наборе стратегий.
- Прогресс показывается в UI.
- Если найдена приемлемая стратегия — применяется.
- Если нет — пользователю даётся выбор: продолжить с лучшей из найденных ИЛИ перейти к ручному выбору.

### Шаг 4. Финализация

- `config.wizard.completed = true` — больше не показывать.
- `config.activeStrategy` — записана выбранная.
- Дашборд открывается, движок запущен.

**Отмена мастера:** пользователь может прервать wizard; в этом случае `wizard.completed` остаётся false, и при следующем запуске мастер откроется снова. До завершения мастера движок не запускается автоматически.

## 8. Обновления: правила

### 8.1. Обновление zapret

**Когда проверять:**

- По кнопке пользователя в UI «Проверить обновления».
- Автоматически раз в сутки при запуске (если включено в config).

**Условие применения:**

- Найденная версия новее текущей.
- Архив прошёл SHA-256.
- Движок можно безопасно остановить (не в момент autoselect).

**Процесс:**

1. Остановить zapret.
2. Backup `zapret/` → `zapret.bak/`.
3. Скачать, проверить, распаковать.
4. Если ошибка — откатиться из backup.
5. Если успех — удалить backup (или хранить один предыдущий).
6. Перезапустить движок с прежней стратегией.
7. Уведомить пользователя.

**Стратегии, отсутствующие в новой версии:**

- Если активная стратегия исчезла из нового zapret — переключиться на ближайший аналог или запустить autoselect.

### 8.2. Обновление ZPUI

**Особенность:** нельзя обновить запущенный `.exe` (файл залочен). Поэтому:

1. Проверка версии (через `cmd/selfupdate` или встроенный checker).
2. Скачивание во временную папку.
3. Запуск `selfupdate.exe` с правами админа.
4. `selfupdate` ждёт завершения текущего ZPUI, подменяет бинарник, перезапускает.
5. Пользователь видит диалог «Применить обновление? Да/Нет».

**Откат:** если после обновления ZPUI крашится на старте — `selfupdate` оставляет предыдущую версию в `ZPUI.exe.bak`, пользователь может вручную восстановить.

## 9. Мониторинг: правила сбора и хранения

### 9.1. Интервал

- Настраивается в config (по умолчанию 30 сек).
- Возможные значения: 15s, 30s, 1m, 5m.
- Слишком частый поллинг даёт ложные картины (DPI «глушит» при всплесках).

### 9.2. Хранение истории

- В SQLite: каждая сэмпл-запись = `{timestamp, resource, available, latencyMs}`.
- Окно хранения: 7 дней сырых данных, 30 дней агрегатов (часовые средние).
- Чистка по расписанию (раз в сутки).

### 9.3. Метрики для UI

- **Текущая доступность** — последнее значение.
- **График за час** — сэмплы за последний час.
- **График за день** — часовые средние.
- **Топ проблемных ресурсов** — ресурсы с наибольшей долей неудач за окно.

## 10. Диагностика: чек-лист

При вызове «Диагностика» (вкладка или тост об ошибке), ZPUI прогоняет набор проверок и выдаёт отчёт с рекомендациями:

| Проверка | Условие | Рекомендация |
|---|---|---|
| Права админа | процесс повышен | «Запустите от имени администратора» |
| WinDivert загружен | драйвер в списке | «Переустановите zapret; проверьте антивирус» |
| Антивирус | детектируется | «Добавьте папку ZPUI в исключения» |
| zapret присутствует | файлы целы | «Переустановите zapret» |
| Стратегия валидна | файл стратегии существует | «Выберите другую стратегию» |
| Целевые ресурсы пингуются | TCP/TLS без обхода | «Провайдер не блокирует — проблема в чём-то ином» |
| С системным DNS | публичные DNS отвечают | «Проверьте настройки DNS» |
| Порт 53 свободен (для DNS) | никто не слушает | «Освободите порт или смените режим» |
| Версия zapret актуальна | последняя | «Обновите zapret» |

Результат: список с зелёными/жёлтыми/красными маркерами и конкретными действиями.

## 12. Журналирование событий

В SQLite `events` пишутся ключевые действия (для истории и отладки):

- `engine.start` / `engine.stop` / `engine.crash` / `engine.restart`
- `strategy.applied` (id, source: manual/wizard/autoswitch)
- `autoselect.run` (winner, bestAvailability, duration)
- `autoswitch.triggered` (from, to, availability)
- `proxy.start` / `proxy.stop`
- `dns.start` / `dns.stop`
- `update.zapret.applied` (from, to)
- `update.zpui.available` (version)
- `wizard.completed`
- `diagnostics.run` (summary)

Срок хранения: 30 дней.

## 13. Пользовательские сценарии (компактно)

Полные сценарии — в [`Концепция.md`](./Концепция.md) §6. Здесь — их формализация в терминах бизнес-логики:

1. **Первый запуск:** Wizard → проверка системы → установка zapret → autoselect → применение.
2. **Ручной старт/стоп:** `engine.start` / `engine.stop`.
3. **Ручная смена стратегии:** `strategy.applied` (source: manual).
4. **Деградация без AutoSwitch:** виден низкий % на дашборде → пользователь открывает диагностику или запускает autoselect вручную.
5. **Деградация с AutoSwitch:** autoswitch triggered → engine restart → `strategy.applied` (source: autoswitch) → notification.
6. **Раздача на телефон:** proxy.start → клиент подключается → трафик идёт через WinDivert.
7. **Раздача на Xbox:** dns.start → консоль использует ZPUI как DNS → корректные ответы для Xbox Live.
8. **Обновление zapret:** update.zapret.applied → движок перезапущен.
9. **Обновление ZPUI:** update.zpui.available → selfupdate → перезапуск процесса.

## 14. Краевые случаи и их обработка

| Случай | Поведение |
|---|---|
| Нет интернета при старте | Wizard не может скачать zapret → сообщение с двумя зеркалами |
| GitHub недоступен | fallback на Яндекс.Диск |
| Антивирус удалил WinDivert | `engine error` → диагностика предлагает переустановить |
| Параллельно запущен второй ZPUI | singleinstance: активировать первое окно, выйти |
| Порт 1080 занят | proxy.error → предложить сменить порт |
| Порт 53 занят | dns.error → инструкция отключить DNS Client |
| Провайдер сменил DPI, стратегии устарели | autoselect не находит > порога → уведомление «обновите zapret» |
| Аккум нет (ноут без сети) | monitor: все ресурсы недоступны, % = 0, но AutoSwitch не сработает (есть smart-проверка на отсутствие сети) |
| Часы системы сбиты | TLS-рукопиатие падает → ложная «блокировка»; диагностика проверяет время |
